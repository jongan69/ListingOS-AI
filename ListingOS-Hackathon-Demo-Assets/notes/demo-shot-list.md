# ListingOS Demo Shot List

## Objective

Capture a clean, credible, fast-moving demo that proves ListingOS is a real seller tool, not a concept mockup.

## Primary Narrative

1. Selling online is still too manual.
2. ListingOS turns product photos into listing-ready eBay output.
3. The seller can keep moving while AI handles the repetitive work.
4. Review is fast, editable, and confidence-aware.
5. The result feels like a real product launch, not a hacky prototype.

## Core Footage Priority

### Must Capture

1. App launch from a clean device home screen.
2. Home screen hero state with brand visible.
3. Photo intake flow.
4. Strategy control or speed-versus-profit choice.
5. Queue / background processing state.
6. Draft review screen with:
   - title
   - price
   - category
   - condition
   - item specifics
   - description
   - confidence / opportunity surface
7. Listing-ready or published proof state.

### High-Value Secondary Clips

1. Smooth scrolling through the review screen.
2. Dragging or reordering photos if available.
3. Polished loading state or queue progress.
4. Inline blocker or verification state if it looks clean.
5. Any premium UI moment that reads instantly on video.

## Capture Order

### Clip A: Problem Hook

- Clean home screen launch
- Hold 1 second on branded hero
- This becomes the first 2 to 3 seconds of the final video

### Clip B: Photos In

- Open picker
- Select product photos
- Confirm selection
- Show that the app immediately transitions into listing flow

### Clip C: AI Working

- Start listing
- Show processing queue or in-progress state
- If the wait is too long, cut to a known-good queued or ready draft

### Clip D: Review Surface

- Open the draft
- Scroll through the review page slowly and steadily
- Pause briefly on each important section

### Clip E: Proof

- Show listing-ready state or published proof
- End on the strongest finished screen with the ListingOS name

## Recording Rules

- Use portrait capture only for the core mobile app clips.
- Keep gestures slow and deliberate enough for viewers to understand.
- Avoid frantic taps.
- Avoid any debug overlays, dev menus, or notification noise.
- If a live publish path is unreliable, stop at listing-ready and use existing published proof as a separate supporting clip.
